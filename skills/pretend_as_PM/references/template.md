# pretend_as_PM — 决策模板与格式规范

本文件包含决策记录模板、汇总报告模板、质量约束和完整示例。由 `pretend_as_PM` 主 Skill 和 `pm-decision`、`pm-summary` wrapper 共同引用。

---

## 一、决策记录模板

每次决策严格按以下格式输出，追加写入项目根目录 `pretend_as_PM.txt`。所有字段不可省略。

```
【决策ID】：PM-YYYYMMDD-HHMMSS-NNNN
【决策时间】：YYYY-MM-DD HH:MM:SS (UTC+8)
【决策重要级】：🔴重要 / 🟡一般 / 🟢轻量
【决策模块】：功能/模块名称（简短，如"Agent意图识别"、"日志采集Pipeline"）
【关联提交】：分支名 + commit hash（如 git 信息可获取；否则标注"未关联"）
【决策背景】：现存问题、原始方案痛点、业务约束、成本/性能/产品侧诉求
            （token开销、执行效率、用户识别准确率、Agent运行稳定性等前置现状）
【备选方案罗列】
方案A：[名称] — 内容简述 + 优点 + 缺点
方案B：[名称] — 内容简述 + 优点 + 缺点
（更多方案依次补充。列出的方案必须是真实可选的。
  不存在有竞争力的替代方案时，诚实说明"本场景无明显替代方案，以下为直接选择"，不硬凑。）
【产品选型结论】：最终选定XX方案，舍弃其余方案
【选型核心依据（PM视角）】
1. 产品业务视角：[如不适用则标"不适用：本决策为纯技术/性能优化"]
2. 大模型/Agent技术特性视角：[如不适用则标"不适用：本决策不涉及模型行为"]
3. 成本&性能视角：[token消耗、推理耗时、资源开销、并发能力等]
4. 长期迭代扩展性视角：[维护成本、新增场景的适配难度等]
   （四个视角按重要性级别填写对应数量。重要级填≥3个，一般级填≥2个，轻量级填≥1个。
    不适用的视角诚实标注，不硬凑。）
【落地收益】：按以下优先级处理 ——
  ① 优先：获取/计算实际数据（如"对比测试：方案A耗时120ms，方案B耗时80ms"）
  ② 其次：基于已知信息估算范围（如"预计token消耗降低10-30%"）
  ③ 最后：无法量化时用具体定性描述（如"消除了意图识别中的冗余规则解析环节，
      减少了单次调用的规则匹配次数"）
  禁止编造精确数字。禁止使用"提升了效率""增强了稳定性"等空泛表述。
【未解决问题】：本次决策中尚未解决的疑问、待确认的假设、遗留风险、需要在后续迭代中关注的边界情况
【后续验证计划】：可执行的验证手段 —— 观察哪些监控指标、观察周期多长、是否需要A/B对比、回归测试范围等
```

## 二、决策ID生成规则

**格式**：`PM-YYYYMMDD-HHMMSS-NNNN`

- `PM`：固定前缀
- `YYYYMMDD`：当前日期（北京时间，UTC+8）
- `HHMMSS`：当前时间（北京时间，24小时制）
- `NNNN`：4位递增序号

**序号生成方式**：
1. 读取项目根目录的 `pretend_as_PM.txt`
2. 文件不存在 → 序号从 `0001` 开始
3. 文件存在 → 找到最后一条 `【决策ID】：PM-` 行，提取序号部分 +1
4. 无法解析上次序号 → 从 `0001` 开始

## 四、质量约束

### 禁止的废话表述

| 禁止 | 替代要求 |
|------|---------|
| "提升用户体验" | 说明具体影响哪个场景/交互/流程 |
| "为未来扩展考虑" | 说明具体什么扩展方向、什么时候可能发生 |
| "更符合最佳实践" | 说明具体什么实践、为什么适用本项目 |
| "提升了效率" | 说明什么效率（开发/执行/响应）、大概幅度 |
| "增强了稳定性" | 说明解决什么不稳定场景、减少什么类型的故障 |
| "优化了性能" | 说明哪个指标（延迟/吞吐/内存）、预期改善幅度 |

### 备选方案原则
- 列出的方案必须是**真实有竞争力**的。不存在明显替代方案时，诚实标注，不硬凑
- **禁止**用"什么都不做""保持现状"作为唯一的对比方案（那是稻草人论证）
- 如果简单方案就是最好的 → 记录为什么简单方案胜出，这就是有价值的决策
- 绝不因为"决策记录需要丰富的备选方案"而引导自己选择复杂方案

### 量化原则
- ① 优先：获取/计算实际数据
- ② 其次：基于已知信息估算范围（用"约X-Y%"，不用虚假精确数字）
- ③ 最后：具体定性描述（说明因果链路，如"减少了一次中间解析，因此单次调用少一个步骤"）
- **禁止**编造精确数字。不确定就说"需上线后观测"

## 五、决策撤回记录模板

当之前记录的决策被推翻或代码回滚时，**不删除或修改已有记录**，新增一条撤回记录：

```
【决策ID】：PM-YYYYMMDD-HHMMSS-NNNN
【决策时间】：YYYY-MM-DD HH:MM:SS (UTC+8)
【决策重要级】：🔄撤回 — 原决策 PM-YYYYMMDD-HHMMSS-NNNN
【决策模块】：与原始决策相同
【关联提交】：当前分支 + commit hash
【决策背景】：撤回原因（新发现的问题 / 验证结果不符合预期 / 外部约束变化等）+ 引用原始决策ID
【备选方案罗列】
原方案：[引用原始决策的选型结论]
新方案：[新选择] — 内容 + 优点 + 缺点
【产品选型结论】：撤回原决策 PM-XXXXXXXXX，改用XX方案
【选型核心依据（PM视角）】：
（按正常要求填写，重点说明为什么原决策不再适用）
【落地收益】：撤回带来的变化
【未解决问题】：新方案引入的新风险
【后续验证计划】：新方案的验证方式
```

## 六、完整示例

```
【决策ID】：PM-20260531-143052-0001
【决策时间】：2026-05-31 14:30:52 (UTC+8)
【决策重要级】：🔴重要
【决策模块】：Agent用户意图识别
【关联提交】：feature/intent-optimize, commit a3f7b12
【决策背景】：
原有Prompt内配置了大量用户意图识别约束规则（约800字），这些规则存在三个问题：
1. 冗余描述拉高单次输入token消耗，按日均500次调用计算，每月额外消耗约120万token
2. 固定规则限制了大模型原生语义理解能力，对非常规/新颖提问易误拦截
3. 新增用户场景时需人工同步更新规则文本，维护成本高
业务约束：意图识别准确率不可下降，误拦截率需保持当前水平以下。

【备选方案罗列】
方案A（保守精简）：保留全部约束规则框架，删减冗余修饰词和重复描述，预计缩减Prompt体积约40%
  — 优点：风险低，保留人工控制力；缺点：根本问题未解决，仍需持续维护规则
方案B（激进放手）：依托大模型原生语义能力，删除全部意图识别类人工限制约束，仅保留基本输出格式要求
  — 优点：Prompt体积大幅缩减（约85%），释放模型原生能力，零维护成本；缺点：存在意图识别漂移风险，需验证无规则约束下的准确率
方案C（混合过渡）：保留核心3条高频场景约束，删除其余低频场景限制
  — 优点：保留关键场景的控制力；缺点：仍保留部分冗余，且保留哪几条的判断本身依赖主观判断

【产品选型结论】：
选用方案B，彻底删除Prompt内意图识别相关人工限制约束，仅保留输出格式要求。

【选型核心依据（PM视角）】
1. 产品业务视角：用户输入意图多样化，固定约束容易拦截合理非常规提问，影响Agent可用性。
   删除约束后，模型可基于语义而非关键词匹配来判断意图，更贴合真实使用场景。
2. 大模型/Agent技术特性视角：当前基座大模型原生语义理解能力已成熟，意图识别类人工约束
   本质上是早期模型能力不足时的冗余兜底，现阶段反而限制了模型原生能力发挥。
3. 成本&性能视角：删除约束文本后Prompt长度从约1200字压缩至约180字，
   预计单次调用输入token降低约85%，月均token费用降低约$XX。同时推理耗时因输入变短而减少。
4. 长期迭代扩展性视角：删除规则后，新增用户场景无需修改Prompt约束文案，
   Agent迭代维护成本趋近于零。后续如需引导模型行为，优先使用few-shot示例而非规则文本。

【落地收益】：
- 单次调用输入token降低约85%（从约1200字降至约180字）
- 月均token费用预计降低相应比例
- 意图识别不再受固定规则限制，对新颖提问的响应更合理
- 后续迭代无需维护意图识别规则文本
- 实际准确率和误拦截率变化需上线后通过A/B测试验证

【未解决问题】：
1. 删除全部约束后，是否存在模型过度解读用户意图的极端情况（如将闲聊误判为任务指令）？
   当前没有充分数据支撑此判断
2. 对于非英语/中文的多语言输入，模型原生意图识别能力是否同等稳定？
3. 如果准确率下降超过可接受阈值（>3%），回滚策略是什么？

【后续验证计划】：
1. 选取过去7天的用户输入（约3500条），用新旧Prompt分别调用模型，对比意图识别结果差异，
   重点关注误拦截率和意图漂移率
2. 上线后灰度发布（10%流量 → 50% → 100%），每个阶段观察至少3天
3. 监控指标：意图识别准确率、用户重新提问率（间接反映意图识别质量）、单次调用平均token消耗
4. 建立回滚机制：如准确率下降超过3%，立即切回方案A的保留核心规则版本
```
