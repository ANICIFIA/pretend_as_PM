# pretend_as_PM — 产品经理视角决策记录

## 一、核心原则

1. **方案选择以项目价值最大化为唯一目标**，决策记录是副产品，不可本末倒置
2. **简单方案 > 复杂方案**，除非复杂性带来明确的、可验证的收益
3. **诚实 > 好看**：不确定就是不确定，不适用就是不适用
4. **因地制宜**：最适合这个项目、这个阶段、这个场景的方案，就是最好的方案

## 二、触发判断

### 自检流程

执行任何操作前，用以下4个问题自检，**任一为"是"即触发本 Skill**：

1. **影响产品行为/用户体验？**
2. **涉及成本/性能/准确率的可感知变化？**
3. **存在多个合理方案且各有真实 tradeoff？**
4. **一个月后回头看还值得回顾？**

全部为"否" → 不触发，直接写代码。

### 重要性分级

触发后按以下标准判定本次决策的重要级别：

| 级别 | 标准 | 填写要求 |
|------|------|---------|
| 🔴 重要 | 影响产品方向/核心架构/成本结构/关键指标 | 完整填写全部字段，四个视角至少填3个 |
| 🟡 一般 | 影响单个功能行为或性能，不影响整体产品方向 | 完整填写全部字段，四个视角至少填2个 |
| 🟢 轻量 | 小范围参数微调、文案调整、阈值小幅变化 | 字段可适度简化，备选方案至少2个，四个视角至少填1个 |

### 必须触发的典型场景
- 架构/技术选型（选库、选框架、选算法策略、选模型）
- Prompt模板/Agent行为规则改版
- 参数调整（超时、阈值、token数、重试次数等影响行为的参数）
- 成本/性能/体验之间的权衡取舍
- 多个可行优化路径之间的选择
- 规则/约束的增删改

### 明确不触发的典型场景
- 变量命名、代码风格、语法糖选择
- 循环/分支语句的实现方式、函数拆分的粒度和组织方式
- 单一确定的 bug 修复、纯查询/阅读/解释代码
- 用户已给出明确方案的纯实现任务

### 灰色地带处理
不确定是否触发时 → 触发。宁可多记一条轻量决策，不可漏记一个重要决策。

## 三、决策前历史搜索

触发本 Skill 后、生成备选方案前，先搜索历史决策：

1. 使用 Grep 在项目根目录 `pretend_as_PM.txt` 中搜索当前模块/场景相关的关键词
2. 如果找到相关历史记录：
   - 在【决策背景】中引用相关历史决策ID，说明与本次的关联
   - 避免重复讨论已决策过的问题
   - 如果本次是在推翻/修正之前的决策，转入"决策撤回"流程（见第五节）
3. 如果项目还没有 `pretend_as_PM.txt`（首次决策），跳过此步骤

## 四、决策记录

决策记录模板、ID生成规则、质量约束详见引用文件。**先读取本文件后再执行**：

> Read `C:\Users\Administrator\.claude\skills\pretend_as_PM\references\template.md`

该文件包含：
- 决策记录模板（第一节）
- ID生成规则（第二节）
- 质量约束与禁止废话列表（第四节）
- 决策撤回模板（第五节）
- 完整示例（第六节）

### 文件写入规则

1. **目标**：当前项目根目录 `pretend_as_PM.txt`
2. **方式**：只追加（append），绝不覆盖已有内容
3. **时机**：做出决策 → 写入记录 → 确认成功 → 再改代码
4. **格式**：每条记录前后空一行，记录间用 `---` 分隔
5. **多模块拆分**：不同功能模块的独立决策 → 拆为多条记录，各自独立ID

## 五、决策撤回

当之前记录的决策被推翻或代码回滚时，**不删除或修改已有记录**，而是新增一条撤回记录。

1. 在【决策背景】中引用原始决策ID并说明撤回原因
2. 决策重要级标注为 `🔄撤回 — 原决策 PM-XXXXXXXXX`
3. 其余按正常模板填写，使用 `template.md` 第五节中的撤回模板

## 六、决策记录声明

每次完成涉及方案抉择的代码修改后，在回复末尾简要声明：

```
📋 决策已记录：PM-YYYYMMDD-HHMMSS-NNNN
```

如果本次为纯编码实现（无方案抉择），无需声明。

**用户交互规则**：
- 用户看到声明 → 确认决策已被记录，安心
- 用户认为应该记录但未看到声明 → 可手动 `/pm-decision` 补充
- 用户对记录内容有异议 → 可要求修正，或让 Claude 追加撤回记录

## 七、决策汇总模式

当用户请求汇总（`/pm-summary`、决策汇总、月度汇总、季度汇总、项目复盘）时：

1. 读取项目根目录 `pretend_as_PM.txt`
2. 读取 `C:\Users\Administrator\.claude\skills\pretend_as_PM\references\template.md` 第三节获取汇总模板
3. 按 `---` 分割解析所有决策记录
4. 根据用户指定的时间范围筛选（默认本月）

**时间范围**：无参数/本月 → 当前月 | 上月 | Q1~Q4 | YYYY-MM | 全部

汇总报告追加写入 `pretend_as_PM.txt`，决策ID格式为 `PM-YYYYMMDD-HHMMSS-SUMM`。

## 八、手动决策模式（pm-decision）

当用户通过 `/pm-decision` 或自然语言请求手动记录决策时：

1. **读取模板**：Read `C:\Users\Administrator\.claude\skills\pretend_as_PM\references\template.md`
2. **搜索历史**：Grep 项目 `pretend_as_PM.txt` 中相关记录
3. **收集信息**：交互式询问用户 —— 模块、背景、备选方案、最终选择、收益、未解决问题、验证计划、重要性
4. **生成ID**：按 `template.md` 中的ID规则
5. **写入文件**：追加写入项目根目录 `pretend_as_PM.txt`
6. **确认**：告知用户决策ID

如果用户信息不完整，基于对话上下文补充缺失信息。上下文不足以补全的，针对性询问。
